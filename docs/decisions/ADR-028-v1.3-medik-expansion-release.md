# ADR-028: v1.3 Release — /medik Expansion + Diagnostic Export + Graphify Measurement Gate

**Deciders**: Ych-Kadmon (architect), arkitect (design), konstruct (implementation plan handoff)

## Context

Kadmon Harness is at v1.2.3. The roadmap `docs/roadmap/v1.3-medik-expansion.md` carries seven open items that together form the narrative for v1.3:

1. `/medik --ALV` diagnostic export (3 MEDIUM items orakle deferred during v1.2.3 `/chekpoint`)
2. `_v: 1` schema field on install-diagnostic entries (forward-compat versioning)
3. Typed reader wrapper for install-diagnostic entries (removes per-caller casts)
4. Three new `/medik` hygiene checks (stale plans, hook health 24h, instinct decay candidates). The roadmap originally listed four; user dropped "unpushed commits" during planning.
5. `.d.ts` filter spec fix in `/medik` Check #6 (code already correct at `ensure-dist.js:43`; docs drift only)
6. `/medik` Check #13 — skill-creator plugin probe (detects missing plugin before `/evolve` step 6 fails)
7. Sprint E graphify adoption measurement gate (committed in ADR-026; must resolve before v1.3.1)

**Discovered during planning**: `/medik` command frontmatter claims "8 health checks", runtime executes 9 (Check #9 Install Health shipped in v1.2.3 without updating the docstring). The roadmap numbered the new checks `#9-12`, colliding with the shipped #9. This ADR locks the renumbering to `#10-13` and documents the drift as a v1.4 hygiene backlog item (a self-assertion check that runtime count matches the frontmatter claim).

This release fits the "one narrative scope" constraint of ADR-025: the binding theme is **`/medik` becomes the hygiene cockpit**, with the diagnostic export and graphify gate as supporting characters. Item 7 is measurement-only — a go/no-go gate that either validates or reverts the graphify adoption committed in ADR-026.

Route A (architecture first) is required because this release introduces (a) a new subsystem directory `scripts/lib/medik-checks/`, (b) a new cross-module type contract `VersionedInstallReport`, and (c) cross-platform redaction logic with a user-controlled input surface (security-sensitive).

## Decisions

### D1 — New subsystem `scripts/lib/medik-checks/` with module-per-check layout

The three new app-layer checks (#10 stale-plans, #11 hook-health-24h, #12 instinct-decay-candidates) plus the probe check (#13 skill-creator-probe) share common patterns (DB access via `getDb()`, filesystem probes, git scanning via `execSync`) but are otherwise orthogonal. Today `/medik` inlines command snippets in `medik.md`; the new checks need TypeScript modules that tests can import.

**Decision**: create `scripts/lib/medik-checks/` with one file per check — `stale-plans.ts`, `hook-health-24h.ts`, `instinct-decay-candidates.ts`, `skill-creator-probe.ts`. Each exports a uniform signature returning `{ status: 'PASS' | 'NOTE' | 'WARN' | 'FAIL', message: string }`.

**Alternatives rejected**:
- *Single `medik-checks.ts` file*: co-locates orthogonal concerns, blocks parallel TDD, fails the < 200-line file preference as soon as all four checks land.
- *Extend existing `scripts/lib/db-health.ts`*: wrong scope — `db-health` is ONE aspect (DB integrity); `medik-checks/` is the category that contains db-health as a peer. Conflating them freezes the wrong abstraction.

**Trade-off**: four new files + one new directory vs. simpler layout. Module-per-check wins on SRP, test isolation, and future scalability (v1.4 hygiene backlog will add more).

### D2 — `VersionedInstallReport` extends, does not wrap, `InstallHealthReport`

Item 3 introduces a typed reader `readTypedInstallDiagnostics()`. The return shape must carry the new `_v` schema field while preserving ergonomic access to existing payload fields.

**Contract**:
```typescript
export interface VersionedInstallReport extends InstallHealthReport {
  readonly _v: number;
}
```

**Forward-compatibility rule**: the reader MUST NOT drop entries missing `_v`. Legacy pre-v1.3 entries remain valid; the reader emits `{ _v: 0, ...entry }` for those at cast time. This is defensive: the log is append-only and rotated, not migrated, so pre-v1.3 entries will coexist with v1.3+ entries for the log's retention window.

**Alternatives rejected**:
- *Discriminated union `type VersionedInstallReport = { _v: 0; ... } | { _v: 1; ... }`*: adds boilerplate at every consumer. Benefit is nil until a v2 schema exists; premature abstraction.
- *Wrap in an envelope `{ version, payload }`*: forces every consumer to destructure; breaks ergonomics with no compensating benefit.

**Trade-off**: extending means any future field addition to `InstallHealthReport` flows into `VersionedInstallReport` automatically. This is the intended behavior — `_v` is metadata ABOUT the payload, not part of it.

### D3 — Cross-platform redaction with literal-string regex, no dependencies

`/medik --ALV` (Item 1) produces a diagnostic file containing paths and git metadata. The redaction logic must strip the real username from home-directory paths across Windows, macOS, and Linux. Inputs come from `os.homedir()`, `process.cwd()`, and file content read from `~/.kadmon/*.log`.

**Security surface**: a user-controlled `cwd` or crafted log content could trigger (a) catastrophic regex backtracking (ReDoS) if the pattern includes user content without escaping, (b) username leakage through edge cases (e.g. path fragments not matching the home-dir prefix exactly).

**Decision**:
- Use **fixed literal prefixes** for platform paths: `/Users/`, `/home/`, `C:\\Users\\`. No user-content interpolation into these patterns.
- Escape `process.cwd()` and `os.homedir()` via an `escapeRegex()` helper before `new RegExp()` construction.
- Write the output file with mode `0o600` (user-only read) as defense-in-depth.
- `spektr` review is MANDATORY in the `/chekpoint` phase for `scripts/lib/medik-alv.ts`.

**Alternatives rejected**:
- *Delegate to a redaction library (e.g. `anonymizer`)*: adds an npm dependency for 80% value (username + path redaction) that simple regex covers. Dependency footprint matters for a harness that ships to multiple projects.
- *Spawn a secondary pass via an external tool*: cross-platform tool availability is uncertain, adds latency, and multiplies the security surface (tool vulnerabilities inherited).

**Trade-off**: simple regex means slightly weaker redaction (we do not strip hostnames, MAC addresses, or internal network identifiers). This is explicitly Out of Scope — the threat model is "accidental username leak when user shares diagnostic with a teammate", not "sanitize for public disclosure".

### D4 — Renumber new checks to #10-13; defer frontmatter-assertion hygiene to v1.4

The roadmap numbered new checks `#9-12` under the assumption that `/medik` had 8 checks. Reality: v1.2.3 shipped Install Health as Check #9 without updating the command frontmatter. Renumber to `#10-13` to avoid collision.

**Decision**: this ADR locks the final numbering:
- #10 — Stale plans (new)
- #11 — Hook health 24h (new)
- #12 — Instinct decay candidates (new)
- #13 — skill-creator plugin probe (new)

The frontmatter `description` field in `.claude/commands/medik.md` bumps from "8 health checks" to "13 health checks" as part of v1.3 release scope.

**Future hygiene (explicitly deferred, NOT v1.3 scope)**: add a `/medik` self-assertion check that compares the runtime check count against the frontmatter claim, failing loudly when they drift. Queued to v1.4 hygiene backlog to avoid scope creep in v1.3.

**Alternative rejected**: renumber retroactively to absorb #9 Install Health under a new category header. This would break any external reference to Check #9 in docs/commits/memory. Additive numbering is append-only and preserves audit trail.

### D5 — Graphify measurement gate uses Opt A (manual `wc` estimate) as primary, Opt B (API simulation) as tiebreaker

ADR-026 committed to a Sprint E measurement gate before v1.3.1: if graphify produces ≥3× token reduction on representative queries, adoption stays; otherwise, rip out. Research-007 identified two methodologies: (A) manual `wc -c` + `bytes/3.5` heuristic, ±30% accuracy; (B) API simulation with real Claude calls, expensive but precise.

**Decision**: run Opt A on 5 recurring architecture queries (captured in the plan). Gate decisions:
- Average ratio ≥ 3.0× → **PASS**. Adoption stays. Promotion to `/medik` Check #14 deferred to v1.3.1 or v1.4 (gate passing is not a license for scope creep).
- Average ratio < 3.0× → **FAIL**. Revert commits `af6e65e`, `7cfd583`, `de3395d`, `1f56014`. Update ADR-026 `status: rejected`.
- Borderline (2.5×–3.5×) → run Opt B on the 2 closest queries. Re-evaluate against the 3.0× threshold.

**Rationale**: Opt A's ±30% accuracy is sufficient for a go/no-go decision outside the 2.5×–3.5× band. ADR-026's exit criterion is binary; precision beyond the threshold wastes API cost.

**Rollback design**: the four graphify commits were structured to be single-commit-reversible — they add files, don't refactor existing ones. Reversion is `git revert` on the four SHAs, update ADR-026 status, done.

### D6 — `/medik` output grouped into four categories (user-locked)

13 checks in a flat table produces scroll fatigue and buries signal. User locked the category grouping during planning:

- **Core** (#1-3): build, typecheck, test
- **Runtime** (#4, 5, 9, 11, 13): live state, DB, install health, hook health, plugin probe
- **Code hygiene** (#6, 7, 8): dist sync, deps, frontmatter
- **Knowledge hygiene** (#10, 12): stale plans, instinct decay

**Alternative rejected**: retain flat table with numeric ordering (historical precedent). Flat table scales poorly — v1.4 hygiene backlog will add more checks, and category grouping is the natural extension point.

**Trade-off**: consumers (human readers, scrapers if any) must learn the category taxonomy. The categories are stable and documented in CLAUDE.md for v1.3+. Scrapers are unlikely — `/medik` output is ephemeral console text, not an API.

## Consequences

### Positive
- `/medik` covers hygiene dimensions it previously ignored: stale plans, hook anomalies, instinct decay, plugin integrity.
- Diagnostic export (`--ALV`) makes support conversations reproducible — user shares one sanitized file instead of a paste scavenger hunt.
- Typed reader + `_v` versioning establishes the forward-compat pattern for any future schema evolution in `~/.kadmon/` logs.
- Graphify gate resolves the deferred ADR-026 uncertainty; whichever way it goes, the ambiguity is closed.
- Category grouping creates the extension point for v1.4 hygiene additions without re-architecting the output.

### Negative
- Seven new files (`scripts/lib/install-diagnostic-reader.ts`, four `medik-checks/*.ts` modules, `scripts/lib/medik-alv.ts`, plus one new `tests/lib/medik-checks/` directory). File count growth is the price of module-per-check discipline.
- Cross-platform redaction test burden: Windows, macOS, Linux path patterns each need dedicated fixtures.
- `spektr` review required for `medik-alv.ts` adds review latency to the final `/chekpoint` full tier.
- Per-item commit cadence (7 feature commits + 1 release commit) extends the release arc; mitigated by TDD per item and reviewed tier footers.

### Risks
| Risk | Mitigation |
|------|------------|
| `--ALV` redaction leaks real username via regex edge case | Cross-platform test suite covering Windows/macOS/Linux username patterns; spektr review MANDATORY; output file mode `0o600`. |
| Check #11 scans huge `hook_events` table on old DBs | `LIMIT 100` + `idx_hook_events_timestamp` (already exists). |
| Check #12 returns noise (pre-v0.5 instincts without `last_observed_at`) | SQL treats NULL as "idle" (`IS NULL OR last_observed_at < ...`). |
| Check #13 false negative if skill-creator is installed in an unusual location | Probe 3 candidate paths: plugin-cache, dev-clone, user scope. |
| Graphify gate returns <3× → scope creep to revert | Revert SHAs pre-identified; ADR-026 designed for single-commit reversal. |
| 13-row output unwieldy | Category grouping (D6) is part of the spec. |
| Commit fatigue across 7 commits | Plan enforces TDD per item + atomic commits to preserve review context. |

## Out of Scope

Explicitly NOT part of v1.3 (protects against scope creep):

- Migrating `/medik` Check #9 to use `readTypedInstallDiagnostics()` — the reader exists; callers migrate gradually across future releases.
- Aggressive `--ALV` redaction (hostnames, MAC-like IDs, internal IPs) — current threat model does not require it.
- `--ALV` output compression — logs are <10 KB; not worth the tradeoff.
- `/medik report` with `gh gist create` integration — revisit if `--ALV` friction proves insufficient.
- Promoting Check #14 `/medik graphify` in the same release when the gate passes — that promotion is v1.3.1 or v1.4.
- `hook-errors.log` automatic rotation — already delegated to `rotating-jsonl-log.ts`.
- Adding `gray-matter` npm dependency for plan-frontmatter parsing — plans use a consistent format; regex suffices.
- Frontmatter-assertion `/medik` check (runtime count vs. docstring claim) — queued for v1.4.

## Rollback

Per-item reversion strategies:

- **D1 (medik-checks/ subsystem)**: delete `scripts/lib/medik-checks/` + corresponding `tests/lib/medik-checks/`; revert `.claude/commands/medik.md` changes referencing the new checks. Single-commit reversal if each check was its own commit.
- **D2 (`VersionedInstallReport`)**: delete `scripts/lib/install-diagnostic-reader.ts` + test; revert `install-diagnostic.js` spread change. Forward-compat rule means pre-revert entries remain readable by post-revert consumers; no migration needed.
- **D3 (redaction in `medik-alv.ts`)**: delete `scripts/lib/medik-alv.ts` + test; remove `--ALV` Phase 0 from `medik.md`. No persistent state to clean up (generated files live in user's `cwd`).
- **D4 (renumber)**: if #10-13 need to be renumbered again, update `medik.md` + CLAUDE.md + CHANGELOG in one commit. Historical git/memory references remain valid because numbering is additive.
- **D5 (graphify gate)**: FAIL path is the rollback itself — `git revert af6e65e 7cfd583 de3395d 1f56014`, update ADR-026 status to `rejected`, update the v1.3 CHANGELOG note accordingly.
- **D6 (category grouping)**: revert the Phase 1 output change in `medik.md` to flat table. Pure presentation change, no data model impact.
