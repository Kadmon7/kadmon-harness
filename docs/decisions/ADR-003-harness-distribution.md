# ADR-003: Harness Distribution Strategy

> **Superseded 2026-04-20 by [ADR-010](ADR-010-harness-distribution-hybrid.md) (partial).** ADR-010 replaced the pure copy-based bootstrap mechanics with a **hybrid plugin + install.sh/ps1 model**. What survived from ADR-003: the copy-based spirit for rules and `permissions.deny` (Claude Code plugins still cannot distribute those two categories, verified again via plan-010 dogfood 2026-04-20). What was replaced: agents, skills, and commands now ship via the Claude Code plugin system using canonical root symlinks (ADR-019), not file copy. Installer scope narrowed to rules + permissions + `.kadmon-version` + user-scope marketplace registration. See ADR-010 §Decision for the 8-angle evaluation and ADR-019 for the symlinks addendum.
>
> **Refresh 2026-04-14:** Status promoted `proposed` → `accepted` after Sprint B/C shipped and almanak re-verified Claude Code plugin system capabilities. Counts in the Distribution Surface table were updated to reflect v1.1 state. The core decision (Option A, copy-based bootstrap) is **unchanged**. See "Refresh Note (2026-04-14)" section at the end of this ADR.

## Context

Kadmon Harness is an operative layer built on top of Claude Code. It is infrastructure, not a product -- built once in `Kadmon-Harness/`, then carried to every project (Kadmon-Sports, ToratNetz, KAIRON, future projects). The harness is v1.1 production-ready with 549 tests passing across 55 files (as of 2026-04-14).

### Build System Dependency

Three lifecycle hooks (`session-start.js`, `session-end-all.js`, `pre-compact-save.js`) dynamically import compiled TypeScript from `dist/scripts/lib/`. This chain:

```
scripts/lib/*.ts  -->  tsc  -->  dist/scripts/lib/*.js  -->  imported by lifecycle hooks
```

Requires `sql.js` and `zod` as runtime dependencies. The `ensure-dist.js` shared module auto-rebuilds dist/ when stale, but the initial `npm install` must happen first.

### The Rules Gap

Claude Code's plugin system supports: agents, commands, skills, hooks (via hooks.json), contexts, MCP servers, and bin scripts. It does **not** support rules. Rules are `.claude/rules/**/*.md` files that are auto-loaded into context based on directory structure. They enforce coding style, security, testing, patterns, performance, git workflow, and agent orchestration -- 19 files totaling the most critical behavioral constraints of the harness.

### The Permissions Gap

`settings.json` contains two sections: `hooks` (distributable via plugin hooks.json) and `permissions.deny` (NOT distributable). The deny rules block dangerous operations:

- Reading `.env` and secrets files
- Force pushing, hard resetting
- Destructive rm commands
- Network tools (wget, nc, ssh, scp)

`settings.local.json` contains `permissions.allow` rules that are machine-specific (user paths, tool allowlists). These must NOT be distributed.

## Decision Drivers

1. **Small team (3-4 people, Mac + Windows)** -- Ych-Kadmon plus one Windows collaborator; two macOS collaborators. Bootstrap must work cross-platform with zero manual OS-specific steps
2. **Two immediate targets** -- ToratNetz (Supabase + pgvector) and KAIRON (React Native + ElevenLabs)
3. **Rules are critical** -- 19 files that enforce quality, security, and workflow. Losing them is unacceptable
4. **Permissions are critical** -- deny rules prevent destructive operations. Losing them is a security regression
5. **Hooks need the build system** -- 3 lifecycle hooks import from dist/, requiring npm install + tsc
6. **Cross-platform hooks** -- hook commands in `settings.json` currently hardcode `PATH="$PATH:/c/Program Files/nodejs"` (Windows-specific). On macOS, Node.js is in the default PATH. The bootstrap script must generate OS-appropriate hook commands
7. **Future-proofing** -- Claude Code's plugin system is maturing (101+ plugins). Locking into a custom solution risks throwaway work if plugins add rules/permissions support later
8. **CLAUDE.md is per-project** -- each project needs its own stack description, agent catalog, pitfalls section
9. **settings.local.json is per-machine** -- each collaborator creates their own allow rules with local paths. Never distributed, never committed

## Options Considered

### Option A: Bootstrap Script (copy-based)

A Node.js script (`scripts/bootstrap.ts`) that copies the harness to a target project directory.

**How it works:**
1. User runs `npx tsx scripts/bootstrap.ts /path/to/target-project`
2. Script copies `.claude/agents/`, `.claude/commands/`, `.claude/skills/`, `.claude/contexts/`, `.claude/rules/`, `.claude/hooks/` to target
3. Script copies `scripts/lib/` to target (TypeScript sources + schema.sql)
4. Script generates `settings.json` for target (hooks config + deny rules), preserving any existing target settings
5. Script merges harness runtime deps (`sql.js`, `zod`) and build deps (`typescript`) into target's `package.json`
6. Script generates a minimal CLAUDE.md template for the target project
7. Script runs `npm install && npm run build` in target
8. Script skips: `settings.local.json`, `agent-memory/`, `dist/`, `node_modules/`, `tests/**` (except `global-teardown.ts`), `docs/` (creates empty scaffolding dirs instead)

**Pros:**
- Distributes EVERYTHING: agents, commands, skills, rules, hooks, permissions, contexts, build system
- Zero gaps: rules and deny permissions are copied directly
- Simple mental model: "copy the harness, then customize"
- Single script, single invocation
- Works immediately on Windows and macOS (Node.js script, not shell script per ADR-002)
- No dependency on Claude Code plugin infrastructure maturity
- Can include a `--diff` mode to show what would change before applying
- Can include an `--update` mode to pull latest harness changes into a target

**Cons:**
- No automatic versioning -- updating requires re-running bootstrap or manual tracking
- Settings merge is fragile: target may have its own hooks, plugins, permissions
- Pollutes target's package.json with harness dependencies (sql.js, zod, typescript)
- Each target has its own copy of 161+ files -- drift is possible if harness evolves
- Two copies of truth: Kadmon-Harness (canonical) and target (copy)
- Manual update process: must re-run bootstrap and resolve conflicts

### Option B: Claude Code Plugin (native packaging)

Package the harness as a Claude Code plugin with `plugin.json` and `marketplace.json`.

**How it works:**
1. Create `plugin.json` declaring agents, commands, skills, contexts
2. Create `hooks.json` declaring all 20 hooks with `${CLAUDE_PLUGIN_ROOT}` paths
3. Use `SessionStart` hook to run `npm install && tsc` in plugin directory
4. Publish to plugin marketplace or use local path reference
5. Target projects enable the plugin in their `settings.json`

**Pros:**
- Native versioning and update mechanism
- Cross-platform: plugin system handles path resolution
- Dependency isolation: harness deps live in plugin directory, not target
- Single source of truth: plugin updates propagate to all projects
- Clean separation: target project's `.claude/` is project-specific only
- Mature ecosystem: 101+ official plugins, proven patterns

**Cons:**
- **Cannot distribute rules/** -- 19 files, the most critical behavioral constraints, are not a plugin component type
- **Cannot distribute permissions.deny** -- security deny rules must be manually replicated in each target
- Workarounds for rules gap are all problematic:
  - Embed in agent prompts: duplication, context waste, fragile
  - SessionStart hook copies rules to target `.claude/rules/`: hacky, file ownership confusion, update conflicts
  - Wait for Claude Code to add rules support: unknown timeline, blocks deployment
- Build system complexity: `SessionStart` hook must run npm install + tsc inside `${CLAUDE_PLUGIN_ROOT}`, first-session latency concern
- Plugin structure requires reorganization of the entire `.claude/` directory

### Option C: Hybrid (Plugin for portable assets + Bootstrap for gaps)

Use a plugin for agents, commands, skills, hooks, contexts. Use a bootstrap script only for rules and permissions.

**How it works:**
1. Package agents, commands, skills, hooks, contexts as a plugin
2. Bootstrap script copies only rules/ and generates settings.json deny rules
3. Plugin's SessionStart hook handles build system (npm install + tsc)
4. Target enables plugin + runs bootstrap once for rules/permissions

**Pros:**
- Gets plugin benefits (versioning, updates) for 80% of the harness
- Rules and permissions are fully distributed via bootstrap
- Clean plugin + minimal bootstrap -- best of both approaches
- Future-proof: when plugins support rules, drop the bootstrap entirely

**Cons:**
- Two distribution mechanisms to maintain and explain
- Two-step install: enable plugin + run bootstrap
- Rules updates require re-running bootstrap (not auto-updated)
- Plugin hooks use `${CLAUDE_PLUGIN_ROOT}` paths but rules reference `.claude/rules/` -- mental model mismatch
- Build system lives in plugin but rules reference patterns that depend on build output
- More complex than either pure approach for a small team

## Decision

**Option A: Bootstrap Script.**

### Rationale

The core constraint is that rules and permissions are non-negotiable. They encode the fundamental behavioral contract of the harness -- the no_context principle, the security deny rules, the coding style enforcement, the agent orchestration chain, the hook catalog. Without them, you get a set of agents and commands that reference rules they cannot access.

Option B (plugin) leaves a 19-file gap in the most critical layer. The workarounds (embedding rules in agent prompts, SessionStart file copying) are worse than the problem they solve. They introduce duplication, fragility, and confusion about which copy of a rule is authoritative.

Option C (hybrid) is the theoretically correct answer, but for a small team with two immediate targets, maintaining two distribution mechanisms adds complexity that outweighs the versioning benefit. The team already manages updates manually across projects -- a `--update` mode on the bootstrap script addresses the 80% case.

**Why not wait for plugin rules support?** The timeline is unknown. ToratNetz and KAIRON need the harness now. Deploying with Option A costs one script (~200 lines of TypeScript). If/when plugins gain rules support, migrating from A to C is straightforward -- the bootstrap script already separates concerns by category.

**Key architectural principles supporting this choice:**
- **ADR-002 (No Bash, No Python):** Bootstrap script is Node.js/TypeScript, consistent with the entire harness
- **Modularity:** Script copies discrete categories, each independently configurable
- **Security-first:** Deny rules are distributed as a first-class concern, not an afterthought
- **Simplicity:** One command, one mechanism, one source of truth (Kadmon-Harness repo)

## Consequences

### Positive

- **Full harness on day one.** `npx tsx scripts/bootstrap.ts ../ToratNetz` gives ToratNetz every agent, command, skill, rule, hook, and permission immediately.
- **CLAUDE.md is templated.** Bootstrap generates a project-specific CLAUDE.md skeleton with correct agent counts, hook references, and environment variables -- but leaves project description, stack, and pitfalls sections for the user to fill.
- **settings.json is smart-merged.** Bootstrap generates hooks config from the canonical hook list, merges deny rules, and preserves any existing target-specific settings (like project-specific plugins or MCP servers).
- **Update path exists.** `--update` mode compares canonical harness files against target copies, shows a diff, and applies changes with user confirmation. This is not automatic versioning, but it is practical for a solo developer.
- **Migration to plugin is non-breaking.** If Claude Code adds rules/permissions to plugins, the bootstrap script can be retired category by category. The file layout in the target project is identical to what a plugin would produce.
- **Cross-platform from day one.** Bootstrap detects the host OS and generates platform-appropriate hook commands in `settings.json`. Mac collaborators get clean `node` invocations; Windows collaborators get the `PATH` prefix. No manual OS-specific steps.
- **Project scaffolding is ready.** `src/`, `tests/`, `docs/decisions/`, `docs/plans/`, `docs/diagnostics/` are created so commands like `/abra-kdabra` and `/medik` work immediately without "directory not found" errors.

### Negative

- **Copy drift.** Target projects can diverge from the canonical harness. Mitigated by `--update` mode and by discipline (do not modify harness files in target projects -- modify in Kadmon-Harness and re-bootstrap).
- **Dependency pollution.** Target's package.json gains `sql.js`, `zod`, and `typescript`. Mitigated by documenting these as harness dependencies and isolating them from project dependencies in a future workspace/monorepo structure.
- **No automatic updates.** User must remember to re-bootstrap after harness changes. Mitigated by a version marker file (`.claude/.kadmon-version`) that session-start can compare against the canonical version.

### Neutral

- **Agent memory is per-project.** Each target starts with empty agent memory. This is correct behavior -- agents should learn project-specific patterns, not inherit Kadmon-Harness patterns.
- **SQLite database is shared.** `~/.kadmon/kadmon.db` already isolates by project hash. No bootstrap action needed for the database.
- **settings.local.json remains manual.** Each collaborator on each machine needs their own allow rules. This is intentional -- allow rules contain machine-specific paths and tool permissions. Bootstrap documents this in the post-bootstrap checklist but never generates or distributes it.

# Install harness to a new project
npx tsx scripts/bootstrap.ts /path/to/target

# Preview what would change

### Settings Merge Strategy

The most delicate part of bootstrap is merging `settings.json`. The target may already have hooks, plugins, or permissions.

**Algorithm:**
1. Read target's existing `settings.json` (or start with `{}`)
2. **Hooks:** Replace entire `hooks` section with canonical harness hooks. Target-specific hooks are not expected because all hooks are generic.
3. **Permissions.deny:** Merge canonical deny rules with existing target deny rules (union, no duplicates)
4. **EnabledPlugins:** Preserve target's existing plugin config. Add harness default plugins if not already present.
5. Write merged result
6. Never touch `settings.local.json`

### Package.json Merge Strategy

1. Read target's existing `package.json`
2. Add harness runtime deps to `dependencies`: `sql.js`, `zod`
3. Add harness build deps to `devDependencies`: `typescript`
4. Add build script: `"build": "tsc && node -e \"require('fs').cpSync('scripts/lib/schema.sql','dist/scripts/lib/schema.sql')\""`
5. Add postinstall: `"postinstall": "npm run build"`
6. Never override existing deps with different version ranges -- warn on conflicts
7. Add `tsconfig.json` include path for `scripts/lib/` if not present

### CLAUDE.md Template

Generated CLAUDE.md includes:
- Quick Start section (npm install + build)
- Core Principle (no_context)
- Environment Variables (KADMON_TEST_DB, KADMON_DISABLED_HOOKS, KADMON_NO_CONTEXT_GUARD)
- Agent table (15 agents with model tiers)
- Command catalog (12 commands by phase)
- Skill catalog (22 skills by domain)
- Hook summary (20 registered + 8 shared)
- Common Pitfalls (DB path, dist/ imports, Windows PATH)
- Placeholder sections for: Stack, MCPs, Project-specific pitfalls

Template is only written if target has no CLAUDE.md. If one exists, bootstrap warns and skips.

### Version Tracking

Bootstrap writes `.claude/.kadmon-version`:
```json
{
  "version": "1.1.0",
  "bootstrapDate": "2026-04-14T00:00:00.000Z",
  "sourceCommit": "f407907",
  "files": 150
}
```

The `session-start.js` hook can compare this against the canonical version in Kadmon-Harness to warn about staleness (future enhancement, not in scope for initial bootstrap).

### Post-Bootstrap Steps (manual)

After running the bootstrap script, each collaborator must:
1. Create `settings.local.json` with machine-specific allow rules (paths, tool allowlists)
2. Customize CLAUDE.md with project-specific stack, MCPs, and pitfalls (first bootstrapper only)
3. Run `npm install && npm run build` (bootstrap runs this, but documenting for awareness)
4. Verify `.gitignore` includes `.claude/settings.local.json` and `.claude/agent-memory/` (bootstrap adds these)
5. Verify hooks work: start a Claude Code session and check session-start output

**For new collaborators joining an already-bootstrapped project:** Steps 2-3 are already done. The new collaborator only needs to create their own `settings.local.json` (step 1) and verify hooks work (step 5).

### Cross-Platform Considerations

**Problem:** All 20 hook commands in `settings.json` currently include `PATH="$PATH:/c/Program Files/nodejs"`, which is Windows-specific. On macOS, Node.js is already in the default PATH, and the `/c/Program Files/` path does not exist.

**Solution:** The bootstrap script detects the host OS at runtime and generates the appropriate hook command template:

```typescript
// Pseudo-code for hook command generation
const isWindows = process.platform === 'win32';
const hookPrefix = isWindows
  ? 'cd "$(git rev-parse --show-toplevel)" && PATH="$PATH:/c/Program Files/nodejs" node'
  : 'cd "$(git rev-parse --show-toplevel)" && node';
```

**Implication for shared repos:** When two collaborators on different OSes share the same repo, they will have different `settings.json` content for hook commands. This is acceptable because:
1. `settings.json` is committed to the repo, so the last bootstrapper's OS wins
2. However, the PATH prefix is harmless on macOS (the `/c/Program Files/nodejs` path simply does not exist, so the PATH append is a no-op)
3. The real risk is a macOS-bootstrapped repo used on Windows, where Node.js would not be found without the PATH prefix

**Recommended approach:** Always use the Windows-compatible hook commands (with the PATH prefix) in the committed `settings.json`. The prefix is a no-op on macOS but required on Windows. This avoids divergence entirely.

**settings.local.json:** Each collaborator creates their own `settings.local.json` with machine-specific allow rules. This file is gitignored and never distributed. Bootstrap should document the required structure in its post-bootstrap output.

### .gitignore Generation

Bootstrap generates or merges a `.gitignore` in the target project with these harness-required entries:

```
node_modules/
dist/
.env
.env.*
*.log
.DS_Store
Thumbs.db
*.db
*.db-journal
.claude.local.md
.claude/settings.local.json
.claude/agent-memory/
```

If the target already has a `.gitignore`, bootstrap appends only the entries that are missing (no duplicates). The `.claude/settings.local.json` entry is critical -- it prevents collaborators from accidentally committing their machine-specific allow rules.

### Future Migration Path (A to C)

When Claude Code plugins support rules and/or permissions:
1. Package agents, commands, skills, hooks, contexts, **rules** as a plugin
2. Reduce bootstrap to only handle permissions.deny (if still not in plugins)
3. If both rules and permissions are in plugins, retire bootstrap entirely
4. Target projects switch from copy-based to plugin-based in one session

The file layout chosen for Option A is intentionally compatible with plugin structure. No reorganization will be needed -- the plugin.json would simply reference the same file paths.

### Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Copy drift in target projects | Medium | Medium | `--update` mode, discipline, `.kadmon-version` staleness check |
| settings.json merge conflicts | Low | High | Smart merge algorithm, `--diff` preview, backup before overwrite |
| Target package.json conflicts | Low | Medium | Never override existing version ranges, warn on conflicts |
| Lifecycle hooks fail on first session | Low | High | `ensure-dist.js` auto-rebuild, bootstrap runs `npm run build` |
| Hook commands diverge across platforms | Medium | High | Bootstrap detects OS and generates platform-appropriate hook commands. Mac users get plain `node` invocations; Windows users get `PATH` prefix. Each collaborator runs bootstrap on their own machine |
| Plugin system adds rules support, making bootstrap partially obsolete | Medium | Low | Migration path is documented. Bootstrap can be retired incrementally |

### Why the decision still holds

The refresh was triggered by a legitimate question: "given that the Claude Code plugin ecosystem has matured (user has 4+ plugins installed from the official marketplace), should we revisit Option B (native plugin) instead of shipping Option A (bootstrap script)?"

almanak verified current Claude Code plugin documentation on 2026-04-14 (sources: code.claude.com/docs/en/plugins-reference, plugin-marketplaces, settings, sub-agents). Findings:

| Requirement | Plugin support (2026-04-14) | Bootstrap support |
|---|---|---|
| `.claude/rules/**/*.md` distribution | **NO** — no `rules` field in plugin.json, no directory convention, not documented | Yes |
| `permissions.deny` merge into target | **PARTIAL** — plugins can ship `settings.json` at root, but merge precedence for plugin-contributed permissions is not documented | Yes |
| `SessionStart` / `Stop` / `PreCompact` hooks fire in plugin context | **NO CONTEXT** — only `SubagentStart` / `SubagentStop` appear in plugin hook docs; lifecycle events for plugins are undocumented | Yes |
| `tsc` / `npm install` build step at plugin install time | **NO** — no install lifecycle event, no postinstall hook; best practice is ship pre-compiled artifacts | Yes |
| Agents, commands, skills, hooks.json (`${CLAUDE_PLUGIN_ROOT}`), mcpServers | **YES** — unchanged from April baseline | Yes |

All three blockers that drove the original ADR-003 decision are **still unresolved**. Specifically:

- **Rules gap is absolute** — no plugin field exists to declare them, no workaround is clean.
- **Permissions merge is ambiguous** — the docs do not guarantee that plugin-contributed `permissions.deny` is enforced additively in the target project. Security-critical rules cannot ride on ambiguity.
- **Lifecycle hooks uncertain** — our 3 lifecycle hooks (`session-start.js`, `session-end-all.js`, `pre-compact-save.js`) import from `dist/scripts/lib/` after `tsc`. Plugins don't document a reliable mechanism to run `tsc` at install time nor to register `SessionStart`/`Stop`/`PreCompact` hooks from within a plugin.

**Conclusion:** Option A (bootstrap script) remains the only mechanism with confirmed end-to-end support for all five Kadmon Harness requirements: rules, permissions.deny, lifecycle hooks, compiled TypeScript with native deps (sql.js, zod), and SQLite schema initialization.

### Future migration trigger

Reopen this ADR if and only if Claude Code plugin documentation adds:

1. A documented `rules` field (or equivalent mechanism) in `plugin.json`, AND
2. An explicit statement that plugin-contributed `permissions.deny` is enforced additively in the target project, AND
3. Documented support for `SessionStart` / `Stop` / `PreCompact` hooks registered by a plugin.

Until all three are present in the official docs, bootstrap (Option A) is the correct distribution strategy.
