## Executive summary

- **Signal-to-noise is low by design**: ~85-90% of the catalog delta is out-of-Kadmon-scope
  padding (13 exotic language packs, community verticals: crypto, healthcare, homelab,
  supply-chain, marketing). Skills hit rate: 12 port-worthy of 216 new (5.5%). Agents/commands:
  ~12 signal files of 111.
- **Learning system: parity holds.** ECC continuous-learning v2.1 (project-scoped instincts,
  cross-project promotion) matches what Kadmon shipped 2026-04-19; Kadmon's decay is something
  ECC still lacks. Post-April delta is operational hardening — see the one exception below.
- **One proven fix for an open P1 bug**: ECC commit `f720885c` "archive observations only after
  successful analysis" is the same failure class as Kadmon's `/forge`-blind-on-long-sessions bug
  (BACKLOG P1, root cause `session-end-all.js` Phase 5 wipe). Their pattern: never clear the
  observations file on Stop; move it to an archive only after the analyzer consumed it.
- **AgentShield**: the scanner source is NOT in the ECC repo (separate repo, npm
  `ecc-agentshield`). Its 102 rules are regex/signature checks over harness config — a good
  config linter with excellent narrative, not a deep analysis engine (no AST taint, 21-entry
  CVE DB, admitted FP cluster). The **concept** — audit the `.claude/` tree as an attack
  surface — is the one layer Kadmon's stack (spektr, blocking hooks, permissions.deny) does not
  cover. Verdict: build the check natively (no external npm dependency); optionally run
  AgentShield once against our repos as third-party validation.
- **ECC moved reviewer agents to read-only tools** (July 2026, `f6e18e08`). Kadmon's reviewers
  carry Write/Edit. Worth a deliberate decision rather than inertia.

## Calibration from Wave 0 (the only shipped evidence)

Wave 0 shipped inside v1.5.0 as `09ea842` — `session-end-all` Phase 5 now archives to
`observations.archive.jsonl` with retain-on-failure, and the three readers (`forge-pipeline.ts`,
`dashboard.ts`, `/kompact` audit) see archive+live. Full-tier `/chekpoint`, 0 BLOCK, suite
1457 → 1462.

What that tells us about the roadmap's judgment:

| Dimension | Roadmap claim | Actual | Verdict |
|---|---|---|---|
| Effort | M | ~M (1 write path, 3 readers, +5 tests) | Accurate |
| Value | "proven fix for an open P1" | Fixed a months-recurring P1 and un-blinded `/kompact`'s own guard | Accurate, arguably understated |
| Risk | not stated | 3 residual items opened in BACKLOG (orphan-recovery read-path parity, O(n^2) archive re-parse, TOCTOU on read→append→unlink) | **Understated** |

**Calibration rule adopted**: an ECC port is not done when the pattern lands — it is done when
every reader/consumer of the changed surface reaches parity. Wave 0 budgeted the write path and
paid for the read paths afterward. Add a parity sweep to the estimate of every remaining item.

**Do not over-generalize Wave 0.** It is the roadmap's best case and does not represent the rest:
it ported a *pattern* into code the harness already owned, against a bug the harness had already
independently root-caused. Waves 2-4 port *new surface area* into a repo that pays a fixed tax per
new component (CLAUDE.md + README + `CATALOG.md` + onboarding catalog + `manifest-schema.test.ts`
`expectedCounts` — the C-001 checklist, itself known-incomplete per BACKLOG P1).

## Trust discount: the roadmap's Kadmon-side claims were not verified

The ECC-side analysis holds up. The **current-state-of-Kadmon** claims did not survive a read of
the live tree:

1. **"Audit the `.claude/` tree as an attack surface — the one layer Kadmon's stack does not
   cover"** — **FALSE.** `.claude/skills/security-scan/SKILL.md` already scans exactly AgentShield's
   five categories (permissions allow-lists, hook command injection, MCP server risk, agent tool
   overreach, CLAUDE.md prompt-injection) and is already declared in `spektr`'s `skills:`
   frontmatter. The gap is *mechanization*, not coverage.
2. **"Kadmon's reviewers carry Write/Edit"** — **FALSE.** `typescript-reviewer`, `python-reviewer`,
   `spektr`, `orakle`, and `kody` all carry exactly `tools: Read, Grep, Glob, Bash`. No Write, no
   Edit. The decision the roadmap asks for was already made.
3. **"/medik unreachable in consumers (9/14 checks)"** — **STALE.** Closed by the 2026-07-12 audit
   waves: `$RUNTIME_ROOT`/`$CONSUMER_CWD` split, per-check consumer guards, ADR-032/033/034, and
   `medik-checks-cli.ts`. Check #9 already reports install health and `install-remediation.ts`
   already emits the doctor/repair narrative.
4. **`skill-scout`** — the function is already covered by `search-first` (codebase → npm/PyPI →
   GitHub) plus `skill-stocktake` (explicitly: "check whether the proposed skill already exists
   under a different name").

**Consequence**: every remaining item MUST be re-verified against the live tree at scheduling
time, not at planning time. Four of the handful of Kadmon-side claims checkable in one pass were
wrong or stale. Note also that `graphify-out/` is ~2.5 months stale (BACKLOG) — it was not, and
must not be, used as the verification source here.

## Verdicts

### Wave 1 — RATIFY with amendments · target v1.6.0 / v1.7.0

| Item | Verdict |
|---|---|
| `/aside` command | **RATIFY.** Confirmed gap — none of the 12 commands does this. |
| kontinuum "WHAT NOT TO RETRY" section | **RATIFY as-is.** Confirmed absent (FREEZE writes Task list / Project summary / Restore steps only). Genuinely ~1h. |
| silent-failure-hunter lens → kody | **RATIFY, but BLOCKED — see amendment A2.** |
| `runtimeConfidence` weighting | **RATIFY, relocated — see amendment A3.** |
| `skill-scout` | **REJECT — see amendment A4.** |
| type-design-analyzer + pr-test-analyzer lenses | **RATIFY as-is.** Fold-ins, no new components. |

- **A1 — "pure prompt, zero deps" understates `/aside`.** Any new component costs a five-surface
  count sync, and C-001's checklist is itself incomplete (BACKLOG P1). Bundle every Wave 1
  component addition into **one** commit so the sync tax is paid once, and close the C-001
  checklist gap in the same pass.
- **A2 — the kody silent-failure lens is sequenced wrong.** kody already flags "empty catch blocks"
  (`kody.md:74`), and the 2026-07-16 silent-swallow pass *established* the `log()` convention
  rather than following one. The open BACKLOG question — does `log()` need a `scrubSecrets()`
  layer, or a documented gate? — is unresolved. Shipping a lens that tells reviewers to add `log()`
  calls before that contract is decided manufactures exactly the "sixth caller" risk spektr
  flagged. **Blocked until the `log()` contract item is closed.**
- **A3 — `runtimeConfidence` belongs in `security-scan`, not spektr's prompt.** spektr already
  loads that skill; the skill has no active-vs-example weighting today. Putting it in the skill
  means `/medik` Check #17 (Wave 3) inherits it instead of re-implementing it.
- **A4 — `skill-scout` rejected.** ~90% covered by `search-first` + `skill-stocktake`. Replace with
  a one-line pre-gate in the skill-creator flow. A 54th skill to solve a documented-elsewhere
  problem is surface for nothing.
- **Narrative split (ADR-025)**: Wave 1 is two narratives, not one. `/aside` + kontinuum =
  operator ergonomics (v1.6.0). Reviewer lenses + `runtimeConfidence` = review quality (v1.7.0).
- **Belongs in v2.0.0?** No. Entirely additive.

### Wave 2 — DEFER · not v2.0.0, not now

- **What it buys the harness**: nothing. It buys ToratNetz and KAIRON. Kadmon-Harness is public,
  project-agnostic infrastructure; these are consumer packs.
- **What it costs**: +3-4 skills, +1 agent (16 → 17), +1-2 rule scopes, each paying the C-001 tax
  and widening the `/medik` Check #8 / #14 drift-audit surface. `react-native-patterns` also
  carries a live-docs dependency (Expo SDK via `/almanak`) that goes stale unattended.
- **The roadmap's "S each, no ADR needed" is wrong.** `rules/react/security.md` introduces a third
  rule scope, and rules auto-load *by file context*. React is a sub-scope of TypeScript, not a
  sibling of it — the routing question (`rules/react/` vs `rules/typescript/react.md`) is
  architectural and unanswered. ADR-035 cut ~11k auto-loaded tokens; re-widening the rule surface
  without deciding the scoping model spends that saving back.
- **Unevaluated alternative that likely wins**: ship consumer packs as an *optional, separately
  installable pack* rather than folding them into core. That preserves ADR-035's token discipline
  and keeps the public harness project-agnostic. **This alternative — not the individual skills —
  is what deserves an ADR** if Wave 2 is ever revived.
- **Trigger to revive**: (a) a ToratNetz or KAIRON session hits a pattern question the harness
  cannot answer, or a collaborator (F.U.S.E / N.O.O.B) asks for it by name; **and** (b) the
  rule-scope routing model is decided. Absent (a), this is speculative inventory.
- **Depends on**: nothing in Waves 1/3/4. Fully parallelizable whenever the trigger fires.
- **Belongs in v2.0.0?** No. Additive whenever it lands — a v1.x MINOR, or out-of-band.

### Wave 3 — RATIFY with major amendments (scope cut ~60%) · target v1.7.0+

| Item | Verdict |
|---|---|
| Harness self-audit check | **RATIFY, reframed** — mechanize `security-scan`, do not build a new layer. |
| Security baseline drift | **RATIFY, sequenced after** the check exists and proves stable output. Small ADR. |
| opensource-sanitizer pipeline | **RATIFY the sanitizer only. REJECT forker + packager.** |
| GateGuard destructive-command gate | **DEFER.** |

- **A5 — the headline item is smaller and better-shaped than described.** `security-scan` already
  covers the five categories; what it cannot do is run unattended or produce comparable output,
  because it is LLM prose. The real work is extracting the deterministic subset into `/medik`
  Check #17 under the established `runCheck(ctx: CheckContext): CheckResult` module pattern
  (`scripts/lib/medik-checks/`, ADR-028/029 precedent). **Effort drops M → S.** The skill stays as
  the judgment layer; the check becomes the mechanical floor. No new agent, no new npm dependency —
  the roadmap's "build native, no external dep" call is **upheld**.
- **A6 — sanitizer only.** The 3-agent pipeline would take the roster 16 → 19. `forker` duplicates
  `gh`; `packager` duplicates `/release`. The **sanitizer** is the only half with no existing owner,
  and it maps to a real standing constraint (harness PUBLIC; Sentinel + Cowork PRIVATE; global
  CLAUDE.md forbids business strategy in this repo). Port it as a **skill**, invoked pre-release.
  Natural trigger: AUD-41 (per-fork upgrade runbook).
- **A7 — GateGuard deferred, and the reason is local.** The harness already runs four blocking
  hooks plus 14 `permissions.deny` entries. AUD-33 is a *live, open* finding that the existing
  `config-protection` heuristic is not a real parser. Adding a second heuristic classifier while
  the first has an open known-incomplete finding is the wrong order — it doubles the maintenance
  surface the roadmap itself warns about (GHSA-4v57, compound-command, ReDoS).
  **Trigger to revive**: AUD-33 resolved or formally closed as won't-fix, **and** a real
  destructive-command incident. Until then `permissions.deny` is the control.
- **Belongs in v2.0.0?** No. Check #17 is a clean MINOR narrative ("/medik gains config-security").

### Wave 4 — SPLIT: one RATIFY, one RATIFY-with-amendment, one DOWNSCOPE, one PROMOTE, one REJECT

| Item | Verdict |
|---|---|
| hookify loop | **RATIFY as the strongest v2.0-class item — but GATED. ADR required.** |
| `/chekpoint` Phase 2 dedup + verifier | **RATIFY with amendment. ADR required.** |
| Manifest install + doctor/repair + CI audit | **DOWNSCOPE** — stated justification is stale. |
| Versioned JSON contracts + schemas | **PROMOTE to Wave 1. No ADR needed.** |
| Reviewers read-only | **REJECT — already true.** |

- **A8 — hookify is right, and it is not ready.** It closes a real named gap (`/forge` learns soft
  patterns; nothing hardens a repeated correction into a block), it has infra precedent
  (`pattern-definitions.json`, ADR-006), and its post-hoc transcript mining correctly sidesteps the
  verified no-user-message-events constraint. But its **inputs are not trustworthy yet**:
  (a) `/forge` was systematically blind on long sessions until Wave 0 shipped **two days ago** —
  there is no clean instinct corpus; (b) `/evolve` is starving (2-3 invocations ever, per the
  2026-07-12 audit); (c) the ECC validation experiment gating `/instinct-import` is **overdue and
  never run**. Building an enforcement layer on a learning layer whose data was truncated for
  months is building on sand. **Gate: >= 4 weeks of post-Wave-0 `/forge` data AND the ECC
  validation experiment closed out.** This is the single largest sequencing change v1.5.0 forces.
- **A9 — dedup yes, "refuted by default" no.** The dedup half is justified by *local* evidence
  stronger than the ECC source: WORK.md's last five `/chekpoint` runs all record duplicate
  cross-reviewer findings ("flagged by spektr + kody", "independently confirmed by kody",
  "surfaced independently by typescript-reviewer"). But an "adversarial verifier that defaults to
  refuted on uncertainty" is a BLOCK-suppression mechanism wearing a different hat, and it collides
  head-on with the consolidator boundary in `rules/common/agents.md` ("kody MAY escalate, MUST
  never downgrade an upstream BLOCK"). **Amendment: the verifier may only append a dissent NOTE,
  never suppress or downgrade a finding** — the same contract kody operates under, mechanically
  checked by the existing Phase 3 dual gate. Also **drop "possibly on the native Workflow tool"**
  until verified via `/almanak`; an unverified platform dependency must not enter a plan.
- **A10 — the installer item's justification no longer holds.** "Directly answers the audit
  headline" is stale (see Trust discount #3). What genuinely remains: (a) a recorded install-state
  artifact and (b) a machine-runnable / CI exit-code mode for `/medik`. Keep those as one S/M item.
  **REJECT the manifest-driven selective installer** — ADR-010 + ADR-021 already decided what ships
  via plugin vs installer; re-opening that without a driver re-litigates a settled decision.
- **A11 — JSON contracts promoted out of Wave 4.** Cheap, additive, and it has a live consumer the
  roadmap did not know about: plan-039's web dashboard shipped two JSON endpoints with Zod on the
  *port only* and no payload schema. It is also the structural fix for the live-repo
  assertion-fragility class (4 recurrences: `8fcd129`, `55c7b6d`, `b3650db`, and the open
  `upgrade-advisory.test.ts` test (d)) — tests pin a versioned contract instead of mutable tree
  state. **No ADR needed**; this is hygiene, not a decision.
- **A12 — reviewers are already read-only.** Verified on disk. Nothing to decide. If anyone wants
  to argue `Bash` is itself a write vector for a reviewer, that is a *new* question with a
  different threat model — not this one, and not a v2.0 blocker.

## Does v2.0.0 still make sense as the vehicle? — NO. Release the reservation.

Test every surviving item against ADR-025's MAJOR criteria (plugin.json schema change · installer
CLI flags/semantics · env-var contract break · removal or rename of a harness-exposed
agent/command/skill/hook · Node engine floor bump):

| Surviving item | Breaks a collaborator's integration? |
|---|---|
| `/aside`, kontinuum section, reviewer lenses, `runtimeConfidence` | No — additive |
| `/medik` Check #17, baseline drift, sanitizer skill | No — additive |
| hookify (new hook + new rule-file format) | No — additive |
| `/chekpoint` Phase 2 dedup + dissent NOTE | No — internal orchestration |
| CI-runnable `/medik` + install-state record | No — additive flags |
| Versioned JSON contracts | No — versioning them is precisely how the break is avoided |

**Zero of them qualify.** Not one renames or removes a public component, alters the manifest
schema, breaks an env var, changes installer semantics, or moves the Node floor.

**Recommendation**: release the v2.0.0 reservation. Ship the surviving waves as sequential
v1.6.0 → v1.9.x MINORs, one narrative each per ADR-025. Reserve v2.0.0 for a change that actually
breaks the public contract — the strongest real candidate is the *other* v2.0 track
(`docs/roadmap/v2.0-multi-project.md`: multi-project + Supabase sync), which does change the
persistence contract and plausibly the env-var and schema surface.

**The reservation has already cost something concrete.** v1.5.0 shipped `kryo` → `kontinuum` — a
textbook ADR-025 MAJOR trigger ("removal or renaming of a harness-exposed ... skill") — as a MINOR,
explicitly because v2.0.0 was being held for this roadmap. That is a policy violation caused by the
reservation, not by the rename. Holding the reservation for a set of additive items guarantees a
second one. Releasing it lets the next genuine break take v2.0.0 honestly.
