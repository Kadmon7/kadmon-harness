# v1.2.4 — Diagnostic Export + Schema Hygiene

> Estado: FUTURO
> Meta: cerrar el loop de "bug → telemetría → compartir" sin depender de copy-paste del colaborador, y consolidar el contrato de `install-diagnostic.log` antes de que tenga consumers múltiples.
> Origen: post-dogfood de v1.2.3 (ADR-024) 2026-04-22 y consolidación de orakle en el `/chekpoint` del sprint install-health (2 MEDIUMs deferidos).

## Scope

Tres items, mismo PR esperado (~2h):

### 1. `/medik --export` — shareable redacted diagnostic file

- [ ] **Subcomando `--export`** en `.claude/commands/medik.md` que produce un archivo `diagnostic-YYYY-MM-DD-HHmm.txt` en el cwd
- [ ] **Contenido agregado**: últimas N entries de `~/.kadmon/install-diagnostic.log` + últimas N de `hook-errors.log` + output fresco de Check #9 + `platform`, `nodeVersion`, `git log -1`, branch
- [ ] **Redaction** antes de escribir: sustituir `os.homedir()` → `~`, rutas absolutas de proyecto → relativas, user-specific paths comunes (`C:\Users\<user>`, `/Users/<user>`, `/home/<user>`) → placeholder
- [ ] **No autenticación, no red** — sólo I/O local. El usuario decide dónde pegar el archivo (DM, GitHub issue, gist manual)
- [ ] **Tests**: redaction cubre los 3 casos (Windows, Mac, Linux) + verifica que el archivo es válido (parseable) + verifica que no contiene el home dir del máquina de test

**Rationale**: middleground entre "copy-paste manual de stderr" (el estado actual, fricción alta) y auto-submit a GitHub (descartado por privacy + spam + requisito de gh auth per collaborator). `/medik --export` = 80 % del valor con 10 % del riesgo. Si después del shipping la fricción sigue (user sigue pegando por DM cada update), upgrade a `/medik report` con `gh gist create` + `gh issue create` consent-gated — Sprint posterior.

### 2. Schema `_v` field en `install-diagnostic.log` entries

- [ ] **`logInstallDiagnostic`** antepone `_v: 1` al serializar el `InstallHealthReport`. Output JSON: `{"_v":1,"rootDir":"...","platform":"...",...}`
- [ ] **`readInstallDiagnostics`** retorna entries con `_v` preservado para que consumers puedan feature-detect
- [ ] **Tests**: verifica que el field está presente en todas las entries escritas post-v1.2.4 y que reader no borra entries legacy sin `_v` (defensive forward-compat)

**Rationale**: orakle MEDIUM (post-`/chekpoint` v1.2.3). Hoy el reader retorna `Array<Record<string, unknown>>` y consumers (hoy: solo `/medik` Check #9 implicit; mañana: el `--export` de arriba) deben defender campo-por-campo. Con `_v: 1` explícito, un futuro upgrade de schema (v1.3.0 agrega campos, v2.0 renombra) puede migrar sin corromper histórico. No añadir esto HOY tiene costo bajo porque `InstallHealthReport` sólo tiene un consumidor (Check #9 latent); añadirlo ANTES de que `/medik --export` llegue es cheap.

### 3. Typed reader wrapper — `readTypedInstallDiagnostics()`

- [ ] **Nuevo export** en `scripts/lib/install-diagnostic-reader.ts` (TS, sibling de install-health.ts): `readTypedInstallDiagnostics(logDir?, limit?): Array<InstallHealthReport & { _v: number }>` — best-effort cast con validación de campos mínimos (`rootDir`, `symlinks`, `timestamp`); entries sin esos campos se omiten con un warning a stderr (o silenciosamente, decidir en implementación)
- [ ] **Consumer**: `/medik --export` lo importa para formatear el diagnostic. `/medik` Check #9 (lectura) lo importa cuando mostremos "últimos N reportes" en la salida
- [ ] **Tests**: fixtures con entries válidas, entries corruptas, entries legacy (sin `_v`), mixto. Verifica que el array retornado tipa correctamente sin casts adicionales en el consumer

**Rationale**: orakle MEDIUM (post-`/chekpoint` v1.2.3). `readInstallDiagnostics` actual retorna `Record<string, unknown>` — aceptable para el productor (session-start) pero penoso para consumers que necesitan acceder `report.symlinks[0].state`. Un wrapper típado 1-shot evita que cada consumer reimplemente el type assertion.

## Orden de implementación sugerido

1. **Item 2 (schema `_v`)** primero — es un cambio trivial (1 línea en `logInstallDiagnostic`, 1 spread extra). Ship solo, tag si aplica.
2. **Item 3 (typed reader)** segundo — depende de item 2 para que el `_v` en `InstallHealthReport` tipado tenga sentido.
3. **Item 1 (`/medik --export`)** último — es el que más valor tiene pero también el más grande (redaction, tests cross-platform, subcomando handling en medik.md).

## Fuera de scope (v1.2.4 específicamente)

- `/medik report` con `gh gist create` + `gh issue create` — reservar para Sprint posterior si `/medik --export` no alcanza
- Redaction más agresiva que paths (ej. anonymize hostnames, MAC-like strings) — sobre-ingeniería para el use case actual
- Compresión del export — los logs son <10KB, no vale complejidad extra
