# v1.1 — Sistema de Aprendizaje Mejorado

> Estado: PLANEADO
> Meta: Que los instincts generen conocimiento promocionable, no solo confirmen que los hooks funcionan
> Prerequisito: Al menos 1 feature completa en ToratNetz usando el harness v1.0
> Referencia: Analisis comparativo ECC vs Kadmon (sesion 2026-04-05)

## Contexto

Los 13 pattern-definitions originales detectaban higiene basica (read before edit, test before commit).
Esto generaba instincts que solo confirmaban que los hooks ya funcionan — no aportaban conocimiento nuevo.
**Actualizado 2026-04-13 (ADR-006)**: Sprint A ejecutado — reemplazamos los 13 hygiene patterns por 12 domain-specific (10 file_sequence + 1 tool_arg_presence + 1 cluster).

Para que el ciclo observe → instinct → skill genere valor real, necesitamos:
1. Patrones domain-specific que detecten comportamiento NO cubierto por hooks
2. Un pipeline que genere skills/commands/agents, no solo recomendaciones
3. Aprendizaje cross-project cuando tengamos 2+ proyectos activos

## Sprint C: DB hardening + observabilidad (mejoras identificadas por reviewers)

Hallazgos de los agentes typescript-reviewer, spektr y orakle durante v0.4 + items de monitoreo migrados desde v1.0:

### DB hardening
- [ ] **[BUG sistemico] `sessions.ended_at` < `started_at`** — diagnosticado 2026-04-13 via `getDbHealthReport()`. **Alcance actualizado**: NO es un caso aislado — **19 de 35 sesiones (54%)** en la DB tienen timestamp inversion. Caso representativo: sesion `2e0d9b68` tiene `started_at=2026-04-12T23:54:40.953Z` y `ended_at=2026-04-12T23:50:33.492Z`. Esto rompe el calculo de duration en el dashboard (muestra valores absurdos como "30h 35m"). Root cause pendiente: `session-end-all.js` podria estar escribiendo `ended_at` con un timestamp de otra fuente (clean-exit.marker? session_start? zona horaria? reloj del sistema al resume?) en vez de `nowISO()` del momento del Stop hook. Reproducir: revisar `updateSession()` en `state-store.ts` y el flujo de `persistSession()` en `session-end-all.js`. Considerar agregar `CHECK (ended_at IS NULL OR ended_at >= started_at)` como guardrail de schema. Prioridad elevada: 54% de corrupcion temporal afecta todas las metricas de duration.
- [ ] **Zod validation en insert boundaries** — validar hookName, eventType, agentType con schemas Zod en insertHookEvent/insertAgentInvocation
- [ ] **ON DELETE CASCADE en FK** — agregar a hook_events, agent_invocations, cost_events (hoy cascade es manual en deleteSession)
- [ ] **CHECK constraints adicionales** — `CHECK (blocked IN (0, 1))`, `CHECK (success IS NULL OR success IN (0, 1))`
- [ ] **LIMIT defensivo en stats queries** — agregar `LIMIT 100` a getHookEventStats y getAgentInvocationStats
- [ ] **SELECT columnas explicitas** — reemplazar `SELECT *` con columnas nombradas (pre-existente en todo state-store)
- [ ] **Dashboard: hook health desde DB** — getHookHealthRows query desde hook_events en lugar de JSONL efimero

### Observabilidad (migrados desde v1.0)
- [ ] **[BLOCKER] `hook_events.duration_ms` siempre NULL** — diagnosticado 2026-04-13 via `scripts/db-health-check.ts`. La columna existe en schema (`schema.sql:74`) pero `logHookEvent()` en `.claude/hooks/scripts/log-hook-event.js` no la esta poblando al escribir a `hook-events.jsonl`, o el extractor en `session-end-all.js` no la esta leyendo al persistir a DB. Efecto: dashboard muestra "—" en Avg ms para todos los hooks, y el item de hook timeout monitoring (abajo) esta bloqueado porque depende de este dato. Root cause pendiente: revisar si `logHookEvent()` recibe el campo desde los 9 hooks que lo llaman, o si se pierde en la extraccion JSONL→DB. Verificacion en DB: `SELECT COUNT(*) FROM hook_events WHERE duration_ms IS NOT NULL` → 0/112.
- [ ] **Hook timeout monitoring** — detectar hooks que exceden el budget de latencia (usar hook_events.duration_ms; alertar cuando >500ms para hooks no-lifecycle). **BLOQUEADO por item anterior** — no sirve alertar sobre una columna vacia.
- [ ] **Validar estimacion de costos** — comparar cost_events estimados con uso real de tokens (requiere telemetria post-uso, cross-check semanal)
- [ ] **Politica de archivado de sesiones** — archivar o borrar sesiones > 90 dias (definir retention policy + comando `/forge archive-old-sessions` o hook de limpieza automatica)

- [ ] **ADR-004 (pendiente arkitect): gate por severidad en /medik** — hoy /medik para en un gate todo-o-nada despues de Phase 2 (diagnostico). Propuesta: auto-aplicar findings LOW/MEDIUM y solo gatear en HIGH/CRITICAL o cuando toquen archivos sensibles (package.json, schema.sql, settings.json). Requiere sesion dedicada de `/abra-kdabra` con arkitect para evaluar trade-offs (gate por severidad vs lockfile vs risk-based) y modificar el skill de /medik. NO escribir el ADR sin arkitect — viola orchestration chain. **NOTA 2026-04-12**: con el sistema tiered /chekpoint ya dogfoodeado, ~80% del valor de este ADR se cubre via tier split (skip/lite/full segun blast radius). Re-evaluar si sigue siendo urgente o si el tier system es suficiente antes de invocar arkitect.

- [ ] **DEP0190 `shell: true` en migrate-v0.4.test.ts** — Node deprecation warning detectada durante el bump de vitest 4. Ubicacion: `tests/lib/migrate-v0.4.test.ts:109-111` usa `spawnSync("npx", [...], { shell: true })`. spektr verifico: inputs son TODOS hardcoded (constante SCRIPT + `os.tmpdir() + Date.now()`), sin input de usuario, sin vector de injection. Severidad: LOW (NOTE). Fix: reemplazar con `execFileSync("npx.cmd", [...], { shell: false })` siguiendo la convencion de hook-testing en `rules/typescript/testing.md`. Hygiene, no urgente.

- [ ] **tsconfig excluir `tests/` del build** — ts-reviewer lo sugirio como cleanup arquitectonico durante el bump de vitest 4. Hoy tsc compila `tests/*.ts` → `dist/tests/*.js` como parte del build pipeline, lo cual infla `dist/` con tests compilados que no sirven en runtime. Arreglamos el sintoma con `exclude: ["dist/**"]` en vitest.config.ts, pero la causa raiz es el tsconfig.json `include`. Requiere verificar que ningun archivo de production importa tipos desde `tests/` (poco probable pero hay que confirmar). Beneficio: dist/ mas pequeño, build mas rapido, menos superficie de confusion. Opcional, v1.1 hygiene.

- [ ] **Limpieza de `$TEMP/kadmon/` (tests)** — diagnosticado 2026-04-13. La carpeta de tmp contiene ~2000+ directorios residuales: `test-mkdir-new-0..999`, `test-isolation-*`, `test-perf-*`, `test-isolate-*`. Estos se crean en tests (parametrized loops con 0..999) pero nunca se limpian. No afecta DB ni runtime, pero ensucia el filesystem y hace ruidosos los listados de `$TEMP/kadmon/`. Fix doble: (a) `afterEach` con `fs.rmSync` en los tests que los crean — buscar con `grep -r "test-mkdir-new\|test-isolation\|test-perf" tests/`; (b) script one-shot `scripts/cleanup-tmp-kadmon.ts` que borre todo excepto directorios de sesiones activas. Hygiene, no urgente.

- [x] **Utilidad `scripts/db-health-check.ts`** — CERRADO 2026-04-13. Promovido a modulo testeable + deteccion de anomalias. Lógica extraida a `scripts/lib/db-health.ts` (`getDbHealthReport(): DbHealthReport`), CLI reducido a thin wrapper (<40 lineas sin SQL inline). Tests en `tests/lib/db-health.test.ts` (5 casos con `:memory:` fixture: empty DB, seeded valid data, timestamp inversion, stale sessions, missing duration). Anomalias detectadas automaticamente: `table_empty`, `sessions_timestamp_inversion`, `hook_duration_missing`, `sessions_stale`. Smoke test contra DB real reporta hallazgos: **19 sessions** con timestamp inversion (no era un caso aislado — es sistemico), 45/45 hook_events en 24h con `duration_ms` NULL. Suite completa: 427/427 tests (422 previos + 5 nuevos). Reuse desde `/medik` Phase 1 diferido como item separado.
