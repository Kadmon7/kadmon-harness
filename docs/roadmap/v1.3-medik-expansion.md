## Scope (7 items)

### Originales de v1.2.4 (3)

#### 1. `/medik --export` — shareable redacted diagnostic file

- [ ] **Subcomando `--export`** en `.claude/commands/medik.md` que produce un archivo `diagnostic-YYYY-MM-DD-HHmm.txt` en el cwd
- [ ] **Contenido agregado**: ultimas N entries de `~/.kadmon/install-diagnostic.log` + ultimas N de `hook-errors.log` + output fresco de Check #9 + `platform`, `nodeVersion`, `git log -1`, branch
- [ ] **Redaction** antes de escribir: sustituir `os.homedir()` → `~`, rutas absolutas de proyecto → relativas, user-specific paths comunes (`C:\Users\<user>`, `/Users/<user>`, `/home/<user>`) → placeholder
- [ ] **No autenticacion, no red** — solo I/O local. El usuario decide donde pegar el archivo (DM, GitHub issue, gist manual)
- [ ] **Tests**: redaction cubre los 3 casos (Windows, Mac, Linux) + verifica que el archivo es valido (parseable) + verifica que no contiene el home dir de la maquina de test

**Rationale**: middleground entre "copy-paste manual de stderr" (estado actual, friccion alta) y auto-submit a GitHub (descartado por privacy + spam + `gh auth` per collaborator). `/medik --export` = 80 % del valor con 10 % del riesgo. Si despues del shipping la friccion sigue, upgrade a `/medik report` con `gh gist create` + `gh issue create` consent-gated — Sprint posterior.

#### 2. Schema `_v` field en `install-diagnostic.log` entries

- [ ] **`logInstallDiagnostic`** antepone `_v: 1` al serializar el `InstallHealthReport`. Output JSON: `{"_v":1,"rootDir":"...","platform":"...",...}`
- [ ] **`readInstallDiagnostics`** retorna entries con `_v` preservado para que consumers puedan feature-detect
- [ ] **Tests**: verifica que el field esta presente en todas las entries escritas post-v1.3 y que reader no borra entries legacy sin `_v` (defensive forward-compat)

**Rationale**: orakle MEDIUM (post-`/chekpoint` v1.2.3). Hoy reader retorna `Array<Record<string, unknown>>` y consumers deben defender campo-por-campo. Con `_v: 1` explicito, un futuro upgrade de schema puede migrar sin corromper historico.

#### 3. Typed reader wrapper — `readTypedInstallDiagnostics()`

- [ ] **Nuevo export** en `scripts/lib/install-diagnostic-reader.ts` (TS, sibling de install-health.ts): `readTypedInstallDiagnostics(logDir?, limit?): Array<InstallHealthReport & { _v: number }>` — best-effort cast con validacion de campos minimos (`rootDir`, `symlinks`, `timestamp`); entries sin esos campos se omiten con un warning a stderr
- [ ] **Consumer**: `/medik --export` lo importa para formatear el diagnostic. `/medik` Check #9 lo importa cuando mostremos "ultimos N reportes" en la salida
- [ ] **Tests**: fixtures con entries validas, corruptas, legacy (sin `_v`), mixto. Verifica que el array retornado tipa correctamente sin casts adicionales en el consumer

**Rationale**: orakle MEDIUM (post-`/chekpoint` v1.2.3). `readInstallDiagnostics` actual retorna `Record<string, unknown>` — aceptable para el productor (session-start) pero penoso para consumers que necesitan acceder `report.symlinks[0].state`. Un wrapper tipado 1-shot evita que cada consumer reimplemente el type assertion.

### Promovidos de v2.0 Hygiene backlog 2026-04-23 (3)

Movidos aqui porque comparten la narrativa `/medik expansion` (ADR-025). Cada uno trae su rationale del origen.

#### 4. `/medik` Checks #9-12 — cobertura de higiene adicional

- [ ] **Check #9: stale plan status** — detectar `docs/plans/*.md` con `status: pending` >3 dias con commits que los referencian. Captura drift "no half-done".
- [ ] **Check #10: unpushed commits** — `git rev-list @{u}..HEAD` + age >24h. Warn si hay production code pendiente de push.
- [ ] **Check #11: hook health 24h** — query a `hook_events` DB por fails/errors recientes (duration_ms > budget, exit_code != 0, blocked). Surface top offenders.
- [ ] **Check #12: instinct decay candidates** — instincts con `confidence < 0.3` y `last_observed_at > 30d` candidatos a archive. Integrar con `decayInstincts()` shipped en plan-018 Phase 1.

**Origen** (v2.0 Hygiene backlog, added 2026-04-17): estimacion ~2-3h total. Motivacion: session-start ya esta pesado (backup + dist rebuild + orphan recovery), no agregar alla — el canal correcto es `/medik` porque se corre pre-milestone. Disparador explicito "post-Sprint-D cleanup antes de v1.2" ya cumplido.

#### 5. `/medik` Check #6 — filtrar `.d.ts` files

- [ ] **Fix**: excluir `*.d.ts` del count en el /medik spec (o en un script helper). El current dist/sync check cuenta `find scripts/lib -name '*.ts'` (18) vs `find dist/scripts/lib -name '*.js'` (17), off-by-one permanente porque `sql.js.d.ts` es declaration file y nunca compila a JS.

**Origen** (v2.0 Hygiene backlog, diagnosticado por mekanik en /medik 2026-04-14 baseline): severidad LOW — no afecta funcionalidad, solo es falso-WARN recurrente. Cabe en scope /medik expansion.

#### 6. `skill-creator` plugin health check

- [ ] **Nuevo Check #13** (o subcheck de #11) — probe del plugin `skill-creator:skill-creator` + fallback claro si missing. Severidad MEDIUM porque el plugin es dependencia critica de `/evolve` step 6 PROMOTE proposals.

**Origen** (v2.0 Hygiene backlog, added 2026-04-17): hoy si el plugin no esta, `/evolve` falla silencioso. El plugin es dependencia critica.

### Nuevo (añadido por user 2026-04-23)

#### 7. graphify — ADOPTED (ADR-026, 2026-04-23)

Spec sourced from upstream repo [safishamsi/graphify](https://github.com/safishamsi/graphify) (MIT, Python 3.10+, 33k stars) + YouTube demo (`docs/research/research-007-graphify-spec-from-youtube.md`). Decision gate resolved to **bucket 3 (external-tool integration)**: graphify is a user-installed Python CLI that writes `graphify-out/` and registers a PreToolUse hook via `graphify claude install`. No harness TS code added.

- [x] **Adoption decision** — 2026-04-23, ADR-026.
- [x] **Narrative fit** — config+docs only. `.graphifyignore`, `.gitignore` entries, README "Using graphify" subsection. Zero internal TS. Single-commit reversible.
- [x] **Harness-side integration shipped** — this roadmap entry, ADR-026, README subsection, `.graphifyignore`.
- [ ] **Measurement gate (Sprint E)** — benchmark 5 typical queries pre/post graphify. Remove adoption if real token reduction < 3×. Promote to first-class `/medik graphify` subcommand (Check #14) if adoption sticks and users commit `graphify-out/` consistently.

**Open risk**: the creator-reported **71.5× token reduction** is not independently benchmarked. Real-world gain may be 3–10×. Sprint E measurement is non-optional before v1.3.1.

## Orden de implementacion sugerido

1. **Item 2 (schema `_v`)** — cambio trivial (1 linea en `logInstallDiagnostic`, 1 spread extra). Prerequisito de items 1 y 3.
2. **Item 3 (typed reader)** — depende de item 2.
3. **Item 5 (/medik Check #6 filter .d.ts)** — fix trivial standalone, sin dependencias. Buen primer win.
4. **Item 4 (Checks #9-12)** — el mas grande en este bloque. Estimacion 2-3h. Cada check es TDD-able por separado.
5. **Item 6 (skill-creator health check)** — depende de item 4 si se implementa como Check #13 integrado.
6. **Item 1 (`/medik --export`)** — el mas grande y el que mas valor tiene. Depende de items 2 y 3 (usa typed reader para formatear). Redaction + tests cross-platform.
7. **Item 7 (graphify)** — requiere spec del user antes de empezar. Puede shippearse aparte si es grande.

## Fuera de scope (v1.3 especificamente)

- `/medik report` con `gh gist create` + `gh issue create` — reservar para Sprint posterior si `/medik --export` no alcanza
- Redaction mas agresiva que paths (ej. anonymize hostnames, MAC-like strings) — sobre-ingenieria para el use case actual
- Compresion del export — los logs son <10KB, no vale complejidad extra
- `hook-errors.log` rotacion — relacionado pero rompe narrativa /medik; queda en v2.0 Hygiene backlog
- ADR-004 gate por severidad en /medik — requiere arkitect/ADR propio; queda en v2.0
