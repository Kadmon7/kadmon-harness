## Eval 1: Hook Latency Benchmark

### Objective
Measure execution time of all 20 hooks against documented budgets from `.claude/rules/common/hooks.md`.

### Budgets
| Tier | Budget | Hooks |
|------|--------|-------|
| Fast | < 50ms | observe-pre, observe-post |
| Medium | < 100ms | no-context-guard |
| Standard | < 500ms | All other 17 hooks |

### Grading
- **PASS** — within budget
- **WARN** — 1-2x budget
- **FAIL** — > 2x budget

### Known Constraint: Node.js Cold Start
Node.js process launch adds ~200-300ms overhead on Windows. This means:
- Raw `node <script>` timing = NODE STARTUP + HOOK LOGIC
- Claude Code may keep Node warm (different runtime behavior)
- We report both **absolute time** and **delta from baseline** (isolating hook logic)

### Approach
Bash script that:
1. Measures baseline cold start: `node -e "process.exit(0)"` x3, take median
2. Runs each hook 3 times with realistic stdin JSON piped via `echo '...' | node <hook>`
3. Takes median of 3 runs per hook
4. Computes delta = median - baseline (approximates hook logic time)
5. Grades against budget using delta

- [ ] Step 1.1: Write `scripts/eval/hook-latency.sh` (S)
  - Contains timing function using `date +%s%N` (nanosecond precision on Git Bash)

  - Verify: script runs and prints baseline timing

- [ ] Step 1.2: Define stdin payloads for each hook category (M)

  - Depends on: 1.1

- [ ] Step 1.3: Run timing loop — 3 runs per hook, compute median and delta (S)

### Phase 2: Verify E2E Matches Unit Tests (S)

### Pattern Detectors (from evaluate-session.js lines 40-69)

**Pattern 1: Read before Edit**
```js
// Adjacent tool_pre pairs where toolSequences[i] === 'Edit' && toolSequences[i-1] === 'Read'
// Threshold: readBeforeEdit >= 3
```

**Pattern 2: Verify before commit**
```js
// tool_pre events where metadata.command includes 'vitest' or 'tsc --noEmit'
// followed by metadata.command including 'git commit' or 'git push'
// Threshold: verifyBeforeCommit >= 1
```


### Approach
Node.js script (`scripts/eval/pattern-detector-eval.mjs`) that:
1. Reads each JSONL file
2. Independently counts patterns using the EXACT same logic as evaluate-session.js
3. Reports raw counts per pattern per session
4. Grades each detector

### Phase 2: Cross-Validate (S)
- [ ] Step 2.1: Run script on all 3 session files (S)
  - Record raw counts: `{readBeforeEdit, verifyBeforeCommit, exploreClusters}` per session
  - Verify: counts are deterministic (running twice gives same result)

- [ ] Step 2.2: Spot-check matches (S)
  - For each pattern with count > 0, print the first 3 matches with line numbers and context
  - Manually verify at least 2 matches per pattern are genuine
  - Verify: no false positives in spot-checked matches
  - Depends on: 2.1

### Phase 3: Assess Detector Quality (S)
- [ ] Step 3.1: Grade each detector (S)
  - **Accurate**: count matches manual spot-check, no false positives or negatives identified
  - **False positive prone**: detector counts things that aren't really the pattern
  - **False negative prone**: manual inspection finds instances the detector missed
  - **Insufficient data**: pattern doesn't appear in any session (can't evaluate)
  - Verify: each pattern has a grade and explanation
  - Depends on: 2.2

### Known Limitations
- **Pattern 1** only checks ADJACENT tool_pre events. If observe-post writes a `tool_post` line between Read and Edit, the `toolSequences` array (which only contains `tool_pre` events) still sees them as adjacent. However, if another `tool_pre` (like a Grep) happens between Read and Edit, it breaks the adjacency. This is a design choice, not a bug — but we should measure how many Read->Edit pairs are broken by intervening tool_pre events.
- **Pattern 2** uses a stateful `hasVerify` flag that resets after each commit. This means if you verify once and commit twice, only the first commit counts. This is correct behavior.
- **Pattern 3** counts clusters, not individual reads. A run of 10 consecutive Reads is 1 cluster, not 3.

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| `date +%s%N` not available in Git Bash on Windows | Eval 1 blocked | Fallback to PowerShell `Measure-Command` or `node -e "performance.now()"` timing |
| Node.js cold start variance too high (> 100ms) to isolate hook logic | Eval 1 results noisy | Report both absolute and delta; run 5 iterations instead of 3 if variance > 20% |
| Lifecycle hooks (session-start etc.) fail DB import | Eval 1 measures error path only | Acceptable — we document this and note that `npm run build` would change timing |
| JSONL files on disk may be from a Windows temp dir that gets cleaned | Eval 3 data loss | Verify files exist before starting; copy to project if needed |
| post-edit-typecheck/quality-gate/post-edit-format are slow by design (shell out to tsc/eslint/prettier) | Eval 1 false FAILs | Grade these separately with a note: "shells out to external tool — budget applies to hook logic only, not child process" |
| `openDb(':memory:')` singleton may leak between Eval 2 tests | Eval 2 state pollution | Explicit `closeDb()` in `afterEach` |
