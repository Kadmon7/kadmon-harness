### Cadena de orquestacion
Cada comando tiene frontmatter que dice que agente invocar y que skills cargar:
```
/code-review → agent: code-reviewer, security-reviewer → skills: coding-standards, receiving-code-review
/kplan → agent: architect, planner → skills: architecture-decision-records
/tdd → agent: tdd-guide → skills: tdd-workflow
```

### Persistencia
- SQLite local (`~/.kadmon/kadmon.db`) con 4 tablas: sessions, instincts, cost_events, sync_queue
- Observations efimeras en JSONL (`/tmp/kadmon/{sid}/observations.jsonl`)
- Orphan recovery: sesiones que crashean se recuperan en el siguiente inicio
- Sessions se cierran limpiamente en compact (no phantom orphans)
- Instincts se evaluan en los 3 escenarios: Stop, Compact, Crash

### Huella de contexto
- 18K tokens siempre cargados (9% de 200K, 1.8% de 1M)
- Arquitectura de 3 niveles: Tier 1 (rules+CLAUDE.md), Tier 2 (TS rules condicionales), Tier 3 (agentes/skills on-demand)
- Espacio de sobra para proyectos encima

## Decisiones clave (ADRs)
| ADR | Que decide | Por que |
|-----|-----------|---------|
| ADR-001 | SQLite local + Supabase planeado | Velocidad local, sync futuro |
| ADR-002 | Solo Node.js en hooks (no Bash/Python) | Consistencia, testabilidad |
| ADR-003 | Un solo perfil de hooks | Simplicidad, evitar conflictos |
| ADR-004 | no-context-guard obligatorio | Prevenir ediciones sin leer primero |
| ADR-005 | Observations efimeras en JSONL | Velocidad (<50ms), no bloquear SQLite |
| ADR-006 | Estrategia de context management | 3 tiers, persistence model, multi-proyecto viable |
