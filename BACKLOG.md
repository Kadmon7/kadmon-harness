## P1 — consistency / quality

- [ ] **`/forge` and `/kompact audit` go blind on long sessions — root cause CONFIRMED live 2026-07-16.** `session-end-all.js:427-442` (Phase 5) `fs.unlinkSync`s `observations.jsonl`, `hook-events.jsonl`, `tool_count.txt` and `last_pre_ts.txt` whenever `messageCount >= 20`. Stop fires at EVERY turn end, so past message 20 every single turn wipes the file. `/forge` reads that ephemeral JSONL rather than the DB, so at the end of a long multi-turn session it only ever sees the last turn. Reproduced by accident during a `/kompact` audit at the end of a ~100+ tool-call session: `wc -l observations.jsonl` returned **6**, zero filePaths. Matches the 2026-07-15 report of `/forge` coming back 100% empty after a dense release+feature session. Was tracked ONLY in the architect's private memory `project_forge_observations_cleanup` — never reached this file, which is why it kept resurfacing as a surprise.
  - **Second victim, worse than the first**: `/kompact`'s own audit step reads the same JSONL, so it reports a 6-call session for a 100-call one. That silently DEFEATS `/kompact`'s safety check — the check says "productive session (>30 tool calls) without /forge -> consider /forge first", the counter reads 6, 6 < 30, so the check never fires on exactly the long sessions it exists to protect. The bug disables its own guard.
  - `session-end-all` already persists agent invocations + hook events to SQLite before the wipe, so the data is not lost — only the reader is pointed at the wrong source. Likely fix: source `/forge` (and `/kompact audit`) from the DB, not the JSONL. Confirm what Phase 5's `messageCount >= 20` condition was protecting against (disk growth?) before changing it — it may be load-bearing.

- [ ] AUD-33 `config-protection.js` residual heuristic scope: `extractBraceBlockAt` is string-literal-aware (Wave 2) but is NOT a full JS/JSON parser — it does not skip JS comments (`/* } */`) or template literals when brace-counting. A disabled rule hidden behind a comment-embedded brace in an `eslint.config.js` could still slip past. Not realistic for the threat model (the "attacker" is the agent/user editing their own config, not a payload crafter; comments aren't valid in `.eslintrc.json`/`tsconfig.json`) — but if we want a true guarantee, replace the regex/scanner approach with `JSON.parse` + object-walk for the JSON configs and a real (lightweight) JS tokenizer for flat-config. Documented as heuristic in the code. Low priority.
